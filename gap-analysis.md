# gap-analysis.md — parity-loop run vs. `libc` 0.2.189 (Linux x86_64)

> Generated by a `cargo public-api` diff of `rusty_libc` (current `main`,
> post-ADR-0003) against the `libc` crate, pinned at `0.2.189`, both on
> `x86_64-unknown-linux-gnu`. This supplements, not replaces, REVIEW.md's four
> hand-curated rounds — see "Method and how this differs from REVIEW.md"
> below for how the two relate.

## Method and how this differs from REVIEW.md

Rounds 1–4 in REVIEW.md were produced by reading libc/POSIX documentation and
`rush`'s actual code by hand. This run instead mechanically diffs the two
crates' public function symbol sets (by bare name, ignoring module path and
signature) to catch anything the manual passes missed. `libc` 0.2.189 on
Linux exposes **3,230** top-level functions; `rusty_libc` currently exposes
**178**. Comparing bare names surfaced **772** libc-only candidates — the
overwhelming majority of which are out of scope for reasons intrinsic to this
crate's design, not case-by-case judgment calls:

- **Not syscalls at all.** `rusty_libc` is a raw-syscall, zero-dependency
  crate with no glibc linkage — anything in `libc` that is pure userspace
  logic rather than a syscall wrapper cannot be "added" here by definition.
  This eliminates most of the list at a stroke: all of `string.h`/`ctype.h`/
  `stdio.h`'s buffered-I/O layer, the malloc/allocator family, `pthread_*`
  (userspace threading primitives — Rust's `std::thread` is the answer, not
  a syscall crate), dynamic linking (`dl*`), locale (`*locale*`, `nl_langinfo`),
  NSS database lookups (`getpwnam`/`getgrnam`/`getspnam`/`getnetby*`/
  `getservby*` and friends — resolved via dlopen'd NSS modules, not a
  syscall), environment variables (`getenv`/`setenv`/pure process-memory
  operations, not syscalls), PRNGs (`rand`/`drand48`/... — `rand::getrandom`
  already covers the one real entropy syscall), calendar/string time
  formatting (`strftime`/`gmtime`/...), regex/glob (`regcomp`/`glob`),
  `getopt`, `backtrace_*`, `iconv`, and System V/POSIX IPC (`msg*`/`sem*`/
  `shm*`/`mq_*` — no known consumer need, same reasoning as REVIEW.md's own
  precedent for declining sockets-adjacent IPC before ADR-0003).
- **Root-only system administration.** `mount`/`umount`/`swapon`/`swapoff`/
  `reboot`/`ioperm`/`iopl`/`personality`/`acct`/`klogctl` — out of scope for
  a job-control shell for the same reason REVIEW.md Round 4 already declined
  `chroot`/namespaces/seccomp.
- **Already declined in REVIEW.md.** `flock` (advisory locking), `getxattr`/
  `setxattr`/`listxattr`/`removexattr`/`f*xattr`/`l*xattr` (extended
  attributes), `sendfile`/`sendfile64`/`splice`/`tee`/`vmsplice`/
  `copy_file_range` — Round 4's "Considered and not proposed" list already
  covers these three families explicitly; re-listing them here would
  contradict, not extend, that decision.
- **Already covered by an existing, differently-shaped primitive.** `vfork`
  (bare) — deliberately not exposed; `vfork_exec`/`vfork_exec_redirected`
  are the safe replacements this crate built specifically because raw
  `vfork` is too dangerous to expose directly (Round 3). `posix_spawn*` —
  glibc's higher-level spawn API, superseded by the same `vfork_exec`
  family. `opendir`/`readdir`/`closedir`/`dirfd`/`rewinddir`/`seekdir`/
  `telldir` — glibc's `DIR*` abstraction over `getdents64`, which
  `fs::dirents`/`fs::getdents64` already wrap directly. `posix_openpt`/
  `grantpt`/`unlockpt`/`ptsname`/`ptsname_r`/`login_tty`/`forkpty` — the
  exact sequence `tty::openpty` already implements as one primitive.
  `alarm`/`getitimer`/`setitimer` — superseded by `timerfd_create` per
  Round 3 item 37's own reasoning. `sigaddset`/`sigdelset`/`sigemptyset`/
  `sigfillset`/`sigismember` — `signal.rs`'s own `u64`-bitmask `sigmask()`
  helper already does this without an opaque `sigset_t`. `strerror`/
  `strerror_r`/`gai_strerror`/`hstrerror` — `Errno`'s own `Display`/`name()`
  already provide this. `execl`/`execle`/`execlp`/`execv`/`execvp`/
  `execvpe`/`fexecve` — variadic/`PATH`-searching convenience wrappers
  around `execve`/`execveat`, which `CStrArray` (the `std`-feature
  `argv`/`envp` builder) already exists to make ergonomic without needing a
  library-side `PATH` search reimplemented in this crate.
- **`accept4` is a false-diff-positive**, not a gap: it's already covered as
  `socket::accept4_in`/`accept4_in6` — the mechanical diff only matches bare
  function *names*, and this crate deliberately names the typed variants
  differently (mirroring `bind_in`/`connect_in`).
- **Superseded by an already-chosen alternative design.** `select`/`pselect`
  (poll/ppoll already the chosen primitive), `sigaltstack`/`sigtimedwait`/
  `sigwait`/`sigwaitinfo` (ADR-0002 makes `signalfd` the primary async-signal
  path; these are the handler-adjacent alternatives it was chosen over).

None of the above got a row below — they didn't survive judgment, matching
the format doc's instruction to list only surviving candidates. What
follows is what's left after all of that filtering: real syscalls, not
previously declined, with a plausible job-control-shell use case.

> **Status:** all 16 candidates below (12 recommended + 4 lower-confidence)
> have been filed as `parity-gap` issues, at the user's explicit request to
> file the full list rather than trim it at this checkpoint.
